# Scientific and Clinical Validation

## Central disclaimer

**SeniorRx Monitor is a research and education project.** No component of this
system has been prospectively validated in a real population, submitted to a
regulatory body (ANVISA/FDA/EMA as Software as a Medical Device — SaMD), or
tested in a clinical trial. The sections below describe *how* the system should
be validated — not a validation that has already been performed.

This validation plan is written to align with recognized international reporting
and appraisal standards, so that any future study is publishable and auditable.

## 1. Validation of the rule engine (Beers / DDI / polypharmacy)

The rule engine is **deterministic and auditable by construction** — every alert
is traceable to a specific row in `beers_pim_criteria` with a cited source.
Validation here is essentially about **implementation fidelity**:

- Unit tests (`tests/unit/`) cover: correct PIM recognition by ATC code, respect
  for the age threshold (≥ 65 years), disease-specific matching by ICD-10,
  pairwise interaction matching, and polypharmacy/eGFR thresholds.
- Before any real-world use, a **clinical test set reviewed by a pharmacist** is
  recommended (a *golden dataset* with known PIM / non-PIM cases), comparing the
  system output against manual review — target metric: **agreement ≥ 95%**, with
  inter-rater reliability reported as **Cohen's κ** (or Gwet's AC1 under high
  prevalence). Ambiguous cases should be adjudicated by a second reviewer.

**Applicable standards:** the reference criteria follow the **2023 AGS Beers
Criteria®**; drug and disease coding follow **WHO ATC/DDD** and **WHO ICD-10**;
the concordance study should be reported per **GRRAS** (Guidelines for Reporting
Reliability and Agreement Studies).

## 2. Validation of the ML model (adverse-event probability)

The label used in `scripts/build_training_set.py` (`adverse_event`) is a
**synthetic latent variable**, generated by a logistic function calibrated to
correlate with PIM count / polypharmacy / low eGFR — **not an observed clinical
outcome**. It exists only to demonstrate the end-to-end MLOps pipeline
(features → training → tracking → serving → monitoring).

**Before any clinical use**, the model must be retrained on a real clinical
label, for example:
- 30-day hospital readmission attributable to an adverse drug reaction (ADR);
- an emergency visit coded for ADR (ICD-10 Y40–Y59 / T36–T50);
- a reported pharmacovigilance event (e.g., WHO VigiBase / ANVISA VigiMed).

**Reporting and appraisal standards.** Any predictive model must be reported
following **TRIPOD+AI** (Transparent Reporting of a multivariable prediction model
for Individual Prognosis Or Diagnosis — AI extension) and appraised for risk of
bias with **PROBAST(+AI)**. Recommended metrics for validation on real data:

- **Discrimination**: AUROC and AUPRC (prioritize AUPRC given the expected outcome
  imbalance);
- **Calibration**: Brier score and a calibration plot (predicted vs. observed
  probability by decile), with calibration slope and intercept;
- **Clinical utility**: decision-curve analysis (net benefit);
- **External validation**: a cohort from a different institution/time period
  before any claim of generalizability (geographic and temporal validation);
- **Fairness**: performance stratified by sex, age band, and care setting to
  detect systematic bias against subgroups (subgroup AUROC and calibration).

## 3. Descriptive epidemiology (R/Quarto report)

`reports/quarto/clinical_report.qmd` produces descriptive statistics (PIM
prevalence, polypharmacy distribution) — useful to characterize a cohort, but it
**does not establish causality**. Observational analyses should be reported per
**STROBE** (Strengthening the Reporting of Observational Studies in Epidemiology).
For causal inference (e.g., "PIM increases falls"), a prospective cohort design or
a **target-trial emulation** would be required, with adjustment for confounding by
indication (e.g., propensity-score or inverse-probability weighting).

This descriptive approach mirrors real published pharmacovigilance work — see, for
example, the author's own cross-sectional study of polypharmacy and PIM in older
adults (Chaves et al., 2020), which reported 65.2% exposure to polypharmacy and
52.2% use of at least one PIM (see [`docs/references.md`](references.md)).

## 4. Data and reproducibility standards

- **FAIR principles** (Findable, Accessible, Interoperable, Reusable) guide the
  data model: standardized coding (ATC, ICD-10), documented schema, and versioned
  datasets (DVC).
- **Reproducibility**: fixed random seeds, pinned dependencies, containerized
  environment, and a single bootstrap pipeline (`scripts/run_pipeline.sh`).
- Any real-data deployment should additionally consider **HL7 FHIR** for
  interoperability and a data-protection assessment under **LGPD/GDPR**.
